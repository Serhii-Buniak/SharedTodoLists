## Архітектурні рішення

### Шарова архітектура

Проєкт побудований за принципом Layered Architecture:

- Api — HTTP API та конфігурація.
- Application — бізнес-логіка, моделі, політики доступу та абстракції.
- Persistence — робота з MongoDB та реалізації репозиторіїв.

Такий підхід дозволяє змінювати транспортний протокол або сховище даних з мінімальними змінами в бізнес-логіці.


### ICurrentUserProvider

Application не має залежати від ASP.NET Core або HTTP.

Для цього поточний користувач отримується через абстракцію `ICurrentUserProvider`.

Поточна реалізація читає `User-Id` із HTTP-заголовка, але за потреби її можна замінити на JWT, Cookie Authentication, OAuth або інше джерело без змін у бізнес-логіці.

### Валідація

Валідація реалізована в Application, а не в API.

Це дозволяє використовувати однакові бізнес-правила незалежно від транспортного протоколу (REST, gRPC, Message Queue тощо).

### Нормалізація

Перед збереженням користувацькі дані нормалізуються (наприклад, обрізаються зайві пробіли), щоб забезпечити консистентність даних.


### Cursor Pagination

Окрім класичної пагінації реалізована cursor-based пагінація.

Для визначення наявності наступної сторінки використовується запит із `limit + 1`, що дозволяє визначити `HasMore` без додаткових запитів до бази даних.

### Docker Compose

Для локального запуску проєкту використовується `docker-compose`, який піднімає необхідну інфраструктуру (MongoDB). Це дозволяє швидко розгорнути середовище без додаткового налаштування.

### GitHub Actions

Налаштовано GitHub Actions для автоматичної перевірки проєкту при кожному push та pull request.

Pipeline виконує:
- відновлення залежностей;
- збірку проєкту;
- запуск тестів.
